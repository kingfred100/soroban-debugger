What It Does

1. Execute and inspect

Run contract entrypoints with JSON args and inspect storage, budgets, events, and stack state.

2. Stop precisely

Set function/source breakpoints, step in/out/over, and review timeline snapshots while paused.

3. Compare behavior

Use symbolic analysis, replay seeds, and benchmark regression checks to catch drift in CI.

Quickstart Demo (CLI)

Basic run with args:

cargo run -- run \
  --contract tests/fixtures/wasm/counter.wasm \
  --function increment \
  --args '[]'

Interactive stepping:

cargo run -- interactive \
  --contract target/wasm32-unknown-unknown/release/contract.wasm

# inside REPL:
break transfer
continue
inspect
storage

Remote server + client:

# terminal A
soroban-debug server --host 127.0.0.1 --port 9229 --token secret

# terminal B
soroban-debug run --remote 127.0.0.1:9229 --token secret

Typical Workflow

  1. Build WASM contract: cargo build --target wasm32-unknown-unknown --release.
  2. Run with deterministic args and optional snapshots to reproduce behavior.
  3. Set source/function breakpoints and inspect storage deltas while paused.
  4. Move to VS Code for source-centric debugging and richer DAP visuals.
  5. Add symbolic and benchmark checks to CI for regression protection.

How To Deploy The VS Code Extension

  1. Prepare accounts and tokens: publisher: soroban in extensions/vscode/package.json, plus a Personal Access Token from Azure DevOps for Marketplace publishing.
  2. Build and validate: cd extensions/vscode && npm ci && npm run ci
  3. Install packaging tool: npm i -g @vscode/vsce
  4. Package locally: vsce package to produce a .vsix artifact.
  5. Smoke test the package: install the generated .vsix in VS Code and run a real debug session against soroban-debug.
  6. Publish: vsce publish patch (or minor/major), then verify listing metadata, changelog, and quickstart instructions.

Quick commands: cd extensions/vscode && npm ci && npm run ci && vsce package && vsce publish patch

Keep extension and CLI versions aligned in release notes, and run DAP smoke tests against the same commit/release line before publishing.
